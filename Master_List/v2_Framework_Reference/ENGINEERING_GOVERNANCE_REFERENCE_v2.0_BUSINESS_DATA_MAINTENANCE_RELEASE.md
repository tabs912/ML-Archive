## Release & Version Governance

| Governance Area | Policy | Migration Guidance | Compatibility Expectations | Approval Workflow |
|---|---|---|---|---|
| Versioning policy | Use vX for production release/milestone, vX.XX for major change, vX.XX.XX for minor correction/cleanup/optimization. Update MASTER_LIST_MERGE_ML_VERSION for generated production scripts. | Preserve previous production release files; document baseline version and rebuild section. | Downstream reports, templates, dashboards, menu callbacks, and public functions remain compatible unless explicitly approved. | Engineering review plus owner approval before release. |
| Migration guidance | Migrations must identify dashboard changes, template changes, worksheet schema changes, public API changes, and archive/timing impacts. | Provide source version, target version, affected sheets, affected templates, validation sequence, rollback notes. | Existing workbooks should continue to load dashboard config and templates without manual repair where possible. | Migration plan approval before production rollout. |
| Compatibility expectations | Public APIs, menu functions, worksheet naming patterns, headers, PMR semantics, and archive behavior are compatibility surfaces. | Use wrappers or transitional aliases for renamed public functions/headers when approved. | No breaking rename/removal without explicit business approval and test evidence. | Business owner approval for public/API/schema breakage. |
| Approval workflow | Changes move through source review, dashboard/template validation, framework health check, timing review, release note approval. | Attach generated governance report updates to release package. | Release cannot be claimed ready while validation reports contain unresolved critical failures. | QA signoff plus framework owner approval. |
| Release evidence | Release package includes production script, Format Dashboard state, template validation, Dashboard Quality Report, Framework Timing Report, configuration and traceability artifacts. | Regenerate docs from production source after final changes. | Evidence must match final committed production source. | Approver verifies version, validation, and release notes. |
